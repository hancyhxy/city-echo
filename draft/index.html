Experiential goal supported — Distributed belonging

Belonging is deposited across the city rather than anchored to a single group or friendship. D

The Parts & How They Connect

Places, sounds, people — and the three screens that link them

City Echo has three kinds of parts — place-parts (Sydney nodes that hold soundscapes), sound-parts (tracks, traces, moods) and people-parts (anonymous users). They connect through a single interaction surface — the map — and three core screens.

Map Discovery mockup

1. Map Discovery

Entry — makes the city's listening activity visible at a glance.

  • Pulse size = number of current listeners at each place
  • Warm amber glow = active soundscape; grey = quiet hour
  • Stylised map, not satellite — emotional geography, not navigation
  • Proximity strip at bottom: one-tap entry to nearest active place
Music Discovery mockup

2. Music Discovery

Dive — lets the user inhabit a single place's soundscape.

  • Mood filter (focus / calm / lonely) changes the playlist — same place, different emotional weather
  • "12 others listening here now" — ambient co-presence line, never a notification
  • Horizontal trace row — Polaroids in a shoebox, not a feed
  • "+ Leave a trace" button always visible, never pulsing — contribution invited, not pressured
Artist Story mockup

3. Artist Story

Deepen — turns a song into a human migration narrative.

  • Bilingual toggle is top-level — language is treated as a user right
  • Hand-drawn journey map visualises displacement, not just biography
  • Artist-curated tracklist links the artist-part to a specific place-part
  • "Leave a trace on her story" — traces can attach to a story, not only a place

How the three screens connect

Map Discovery → Music Discovery → Artist Story

Zoom → in → deeper. Users always return to the map. Every pin on the collage opens to a Music Discovery screen; every trace tag was left from a Music Discovery screen; every artist-polaroid is accessible through an Artist Story screen.

Asymmetry of identity: artists are named; users are not. Named parts: places, artists. Anonymous parts: users (avatar + one-word traces only). This asymmetry is the connection rule that makes the whole system emotionally safe.

E

System (Technical) Blueprint

Components, technologies, hierarchy of interfaces

City Echo is organised into five layers, with inclusion infrastructure running as a cross-cutting band spanning all four upper layers. Three design principles constrain the stack: ambient and low-pressure, anonymous by default, and place-first, not user-first.

Layer 1 · Presentation

User-facing interfaces

  • iOS app — SwiftUI, iOS 17+
  • Android app — Jetpack Compose, Kotlin
  • Web companion — Next.js 15 (Artist Story deep-link pages)
  • Ambient widget — home-screen "Near you" glance
Layer 2 · Device services

On-device logic

  • Location — CoreLocation, FusedLocationProvider, geofence API
  • Audio playback — Spotify SDK, Apple MusicKit, 30s preview fallback
  • On-device trace composer — offline-capable, syncs on reconnect
  • Ambient haptics — CoreHaptics / VibrationEffect (never intrusive)
  • Local cache — Core Data / Room for offline commute use
Layer 3 · Application services

Backend APIs · Node.js + NestJS · AWS ECS Fargate

  • Soundscape service — playlist + mood distribution + listener count per place
  • Trace service — read/write traces, reactions, moderation hand-off
  • Place registry — curated catalogue of recognised places + geofence defs
  • Artist Story service — bilingual content, curated tracklists
  • Identity service — anonymous accounts (avatar + language + region)
  • Moderation service — automated + human queue for traces & tags
  • Live presence service — WebSocket "N listening now" counters
Layer 4 · Data & integrations

Stores and external APIs

  • PostgreSQL 16 (AWS RDS) — places, users, traces, stories
  • Redis (ElastiCache) — live counters, WebSocket sessions
  • OpenSearch — moderation dashboard search
  • AWS S3 — story photos, hand-drawn journey maps, avatar pack
  • Spotify & Apple Music APIs — catalogue + playback rights
  • Mapbox — stylised tiles + geocoding (chosen for amber styling)
  • PostHog (self-hosted) — privacy-preserving analytics
Layer 5 · Inclusion infrastructure · cross-cutting

Culturally and linguistically safe substrate

  • Bilingual content pipeline — every story, tag, UI string stored with language variants
  • "International students' picks" seed dataset — prevents cold-start bias toward Anglo-Australian indie
  • Moderation pipeline — three-stage: automated → cultural-sensitivity → human review by trained community moderators
  • Language-first onboarding — language picked before region; UI defaults to chosen language, not device locale
  • Accessibility layer — VoiceOver / TalkBack audits, no text-in-image

Hierarchy of interfaces (rubric requirement)

  1. Tier 1 — Primary: Map Discovery (home screen on mobile)
  2. Tier 2 — Situated: Music Discovery (place-scoped) · Artist Story (song/story-scoped)
  3. Tier 3 — Creation: Trace Composer
  4. Tier 4 — Ambient: Home-screen widget · Lockscreen live-activity · Rate-limited push (max 1/place/day)
  5. Tier 5 — Share-out: Web Artist Story (read-only public URL) · Deep-links for WeChat / WhatsApp / email

Rule: Users can always return to Tier 1. Tiers 4 & 5 do not require the app to be open.

What the blueprint deliberately omits

  • No DMs or chat — breaks ambient, low-pressure
  • No friend / follow graph — breaks anonymous by default
  • No algorithmic For-You feed — breaks place-first; the map is the feed
  • No user profile pages — traces are the contribution unit, not identities
  • No leaderboards — gamification would reintroduce the social-performance pressure the design exists to remove

Absences are design decisions. The blueprint is as much about what we refuse to build as what we do build.
